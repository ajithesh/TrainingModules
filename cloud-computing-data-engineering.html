📌 What is Cloud Computing?

▼

The foundation of modern IT — delivering computing resources over the internet

Definition On-demand IT resources via the internet Model Pay-as-you-go Access Anywhere, anytime Managed by Cloud Service Providers

Teaching hook: "Imagine you need electricity at home. You don't build your own power plant — you plug into the grid and pay for what you use. Cloud computing is the same idea, but for IT resources like servers, storage, and databases."

Cloud Computing is the delivery of computing services — including servers, storage, databases, networking, software, analytics, and AI — over the internet ("the cloud") to offer faster innovation, flexible resources, and economies of scale.

Before Cloud vs. After Cloud

🏢 Before Cloud (On-Premises / Traditional IT)

  • Companies bought physical servers and stored them in their own data centers
  • Needed a dedicated IT team to maintain hardware 24/7
  • Had to estimate capacity months in advance — buy too much = waste money, buy too little = system crashes
  • Scaling up meant ordering new hardware, waiting weeks for delivery, and setting it up manually
  • Disaster recovery required a second data center (very expensive)

Like owning a car — you pay for it whether you drive it daily or it sits in the garage.

☁️ After Cloud

  • Rent servers, storage, databases from a cloud provider (Azure, AWS, GCP)
  • The provider maintains all the hardware — you focus on your application
  • Scale instantly — need more power? Add it in minutes. Need less? Scale down and stop paying
  • Pay only for what you actually use (like a utility bill)
  • Built-in disaster recovery — your data is replicated across multiple locations automatically

Like using Uber — you pay only when you ride, no parking fees, no maintenance.

The Simple Analogy

🏠 On-Premises

You build your own house. You buy the land, materials, hire workers, handle plumbing, electricity, maintenance — everything is your responsibility.

Full control, but full cost and full headache.

🏨 Cloud Computing

You stay in a hotel. Someone else built it, maintains it, provides electricity, water, WiFi. You just check in and use it.

Less control, but way less hassle. Pay per night.

🏘️ Hybrid

You own a house but also book hotels when traveling. Keep sensitive stuff at home, use the hotel for convenience when needed.

Best of both worlds — many enterprises use this model.

💰 Pay-As-You-Go Model

Cloud computing follows a Pay-As-You-Go (PAYG) pricing model — you only pay for the resources you actually consume, similar to how you pay for electricity or water.

  • Used a VM for 3 hours? Pay for 3 hours only.
  • Stored 50 GB of data? Pay for 50 GB only.
  • Ran 1,000 function executions? Pay for 1,000 only.
  • Didn't use anything this month? Pay ₹0.

No upfront payment. No long-term contracts (unless you choose one for discounts). Start small, scale big.

💼 CapEx vs. OpEx — Cloud is OpEx!

💡 Key concept: One of the biggest financial benefits of cloud computing is shifting from CapEx to OpEx. This is an important interview topic!

🏗️ CapEx — Capital Expenditure

Spending money upfront on physical infrastructure that you own.

  • Buying servers, networking equipment, data center space
  • Big one-time cost — spend ₹50 lakhs today, use it for 5 years
  • Value depreciates over time (hardware gets old)
  • You're stuck with what you bought — even if you don't need it all
  • Requires upfront planning — guess how much you'll need in 3-5 years

Analogy: Buying a car — big upfront cost, insurance, maintenance, depreciates every year.

Examples: On-premises servers, data centers, network switches, UPS systems

☁️ OpEx — Operational Expenditure ⭐ Cloud!

Spending money on services as you use them — no ownership, just consumption.

  • Renting servers, storage, databases from a cloud provider
  • Small recurring costs — pay ₹10,000/month, stop anytime
  • No depreciation — you don't own anything
  • Flexible — scale up in minutes, scale down when not needed
  • No guessing — use what you need today, adjust tomorrow

Analogy: Taking Uber/Ola — pay per ride, no maintenance, no parking, cancel anytime.

Examples: Azure VMs, AWS EC2, cloud storage, SaaS subscriptions (Office 365)

🤔 Why is Cloud Computing Required?

▼

💡 Key question to ask learners: "What would you do if your app suddenly gets 10x more users overnight? Buy 10 more servers? Cloud solves exactly this."

The Problems Cloud Computing Solves

💰 1. High Upfront Costs

Problem: Setting up a data center costs lakhs/crores — servers, cooling, power, networking, security, real estate.

Cloud solution: Zero upfront cost. Start with ₹0 and pay monthly for what you use. A startup can launch on the same infrastructure as a Fortune 500 company.

📈 2. Unpredictable Growth

Problem: You can't predict traffic. An e-commerce site might get 50 users today and 50,000 during a sale.

Cloud solution: Auto-scaling — the cloud automatically adds more servers during peak traffic and removes them when traffic drops. You never over-provision or under-provision.

🌍 3. Global Reach

Problem: If your server is in India and users are in the USA, they experience slow loading times (latency).

Cloud solution: Cloud providers have data centers in 60+ regions worldwide. Deploy your app closer to your users with a few clicks.

🛡️ 4. Security & Compliance

Problem: Securing your own data center against hackers, natural disasters, and data breaches is extremely hard and expensive.

Cloud solution: Cloud providers invest billions in security — encryption, firewalls, identity management, compliance certifications (ISO, SOC, HIPAA, GDPR).

⚡ 5. Speed of Innovation

Problem: In traditional IT, setting up a new environment takes weeks — procurement, installation, configuration.

Cloud solution: Spin up a new server in seconds. Need a database? Click a button. Need AI? It's an API call away. Teams can experiment and iterate faster.

🔧 6. Maintenance Burden

Problem: Your IT team spends 70% of their time on maintenance — patching, upgrades, hardware failures — instead of building new features.

Cloud solution: The cloud provider handles all infrastructure maintenance. Your team focuses on what matters — building your product.

💰 Cost EfficiencyNo hardware to buy. Pay-as-you-go model. Convert CapEx (capital expense) to OpEx (operational expense).
📈 ScalabilityScale up/down in minutes. Handle 10 users or 10 million users with the same setup.
🌍 Global ReachDeploy to any region in the world. Reduce latency for global users.
⚡ Speed & AgilityProvision resources in seconds. No waiting for hardware procurement.
🛡️ SecurityEnterprise-grade security, encryption, compliance certifications. Providers spend billions on security R&D.
🔄 Disaster RecoveryBuilt-in backup and replication across regions. Your data survives even if a data center goes down.

❌ Cons (Challenges)

🌐 Internet DependencyNo internet = no access to your cloud resources. Critical for remote areas with poor connectivity.

📉 Limited ControlYou don't control the underlying hardware. Outages at the provider level affect you (rare but happens).
📚 Learning CurveCloud platforms have hundreds of services. Choosing the right one and configuring it correctly requires expertise.
⏱️ LatencyFor ultra-low-latency applications (real-time gaming, trading), cloud may add slight delays vs. on-prem.
🏛️ Compliance ComplexityDifferent industries/countries have different rules about where and how data can be stored.

🧱 Service Models — IaaS, PaaS, SaaS

▼

Teaching hook: "Think of it like pizza. You can make it from scratch (On-Prem), buy the dough and add toppings (IaaS), get it pre-made and just bake (PaaS), or order delivery (SaaS)."

The Pizza Analogy 🍕

🏠 On-Premises (Make at Home)

You do everything — grow wheat, make dough, make sauce, bake, serve. Total control, total effort.

IT equivalent: You buy servers, install OS, configure networking, deploy apps, manage security — everything is yours.

🖥️ IaaS — Infrastructure as a Service (Take & Bake)

Someone gives you the kitchen, oven, and raw ingredients. You still cook and serve, but you don't build the kitchen.

IT equivalent: Cloud gives you VMs, storage, networking. You manage the OS, runtime, and applications.

Azure example: Azure Virtual Machines, Azure Disk Storage

AWS example: EC2, EBS

⚙️ PaaS — Platform as a Service (Pizza Kit Delivery)

Someone gives you a ready-to-bake pizza kit — dough, sauce, cheese. You just add toppings and bake.

IT equivalent: Cloud gives you the platform (OS, runtime, middleware). You just deploy your code. No server management.

Azure example: Azure App Service, Azure SQL Database, Azure Functions

AWS example: Elastic Beanstalk, RDS, Lambda

📦 SaaS — Software as a Service (Pizza Delivery)

Someone delivers a fully cooked pizza to your door. You just eat it. No cooking, no dishes.

IT equivalent: The cloud provides the entire application. You just use it — login, work, done.

Examples: Microsoft 365, Gmail, Salesforce, Zoom, Dropbox, Teams

Choose IaaS when...

  • You need full control over the OS
  • Running legacy apps that need specific configurations
  • Lift-and-shift migration from on-prem

Choose PaaS when...

  • You want to focus on code, not infrastructure
  • Building new web apps, APIs, or microservices
  • You want auto-scaling and managed updates

Choose SaaS when...

  • You need ready-to-use software
  • Email, CRM, collaboration, productivity tools
  • No development needed — just use it

🏗️ Cloud Deployment Models

▼

Cloud deployment models define where the cloud infrastructure lives and who can access it.

☁️ Public Cloud

Resources are owned and operated by a third-party provider and shared across multiple organizations (multi-tenant).

  • Who uses it: Startups, small businesses, anyone who wants to avoid hardware costs
  • Pros: Cheapest option, no maintenance, instant scaling
  • Cons: Less control, shared infrastructure
  • Examples: Azure, AWS, Google Cloud

Analogy: Staying in a hotel — shared building, you rent a room.

🏢 Private Cloud

Cloud infrastructure dedicated to a single organization. Can be hosted on-premises or by a third party.

  • Who uses it: Banks, government, healthcare — anyone with strict security/compliance needs
  • Pros: Full control, enhanced security, meets compliance requirements
  • Cons: Expensive, requires in-house expertise
  • Examples: Azure Stack Hub, VMware Private Cloud, OpenStack

Analogy: Owning a private villa — all yours, but you maintain it.

🏘️ Hybrid Cloud

Combines public and private clouds, allowing data and applications to move between them.

  • Who uses it: Enterprises with legacy systems + new cloud initiatives
  • Pros: Flexibility, keep sensitive data on-prem, burst to public cloud for scale
  • Cons: Complex to manage, needs integration expertise
  • Examples: Azure Arc, AWS Outposts, Google Anthos

Analogy: Own a house + book hotels when traveling.

🌐 Multi-Cloud

Using multiple public cloud providers simultaneously (e.g., Azure + AWS) to avoid vendor lock-in.

  • Who uses it: Large enterprises that want redundancy and best-of-breed services
  • Pros: No vendor lock-in, pick best services from each provider
  • Cons: Most complex, requires multi-cloud expertise
  • Examples: Using Azure for AI + AWS for storage + GCP for data analytics

Analogy: Shopping at multiple stores — fruits from one, dairy from another.

🔑 1. Microsoft Entra ID (formerly Azure Active Directory)

💡 Simple definition: Microsoft Entra ID is Azure's identity and access management service. It answers one fundamental question: "Who are you, and what are you allowed to do?"

🏢 What is a Tenant?

When your organization signs up for Azure, it gets a tenant — a dedicated instance of Entra ID. Think of it as your company's identity universe.

  • Every tenant has a unique ID (Tenant ID / Directory ID)
  • All users, groups, and apps in your organization live inside this tenant
  • One tenant can have multiple subscriptions

Analogy: The tenant is like your company's ID card system — it knows everyone who works there.

� Tenant ID & Primary Domain

Every tenant has two key identifiers:

Tenant IDA unique GUID (globally unique identifier) assigned to your tenant. It never changes. Example:
a1b2c3d4-e5f6-7890-abcd-ef1234567890
Primary DomainWhen you first create a tenant, you get a default domain:
yourcompany.onmicrosoft.com
This is your primary domain — it's free and permanent.
Custom DomainYou can add your own domain (e.g., techcorp.com) and verify it via DNS. Users then login with ajay@techcorp.com instead of ajay@techcorp.onmicrosoft.com.

👤 Adding Users — Domain Auto-Attaches

When you create a user in Entra ID, the domain automatically gets attached to their username:

// You create a user with just the name "ajay"
// Entra ID automatically makes it:

Default domain:  ajay@techcorp.onmicrosoft.com
Custom domain:   ajay@techcorp.com  (if custom domain is added)

// This is their UPN (User Principal Name) — their login ID
// Every user MUST have a domain suffix — you can't create "ajay" without one

Key points:

  • The domain suffix is automatic — you pick the username, the domain is appended
  • If you have multiple domains (e.g., techcorp.com + techcorp.onmicrosoft.com), you choose which one to use for each user
  • This username + domain = UPN (User Principal Name) — it's how you log into Azure, Office 365, Teams, everything
  • Guest users from other organizations keep their own domain (e.g., partner@othercorp.com) — they're invited, not created

Analogy: It's like a company email — when HR adds "Ajay" as an employee, he automatically gets ajay@company.com. He doesn't pick the domain part.

💳 2. Azure Subscriptions

💡 Simple definition: A subscription is a billing and access boundary in Azure. It's like a credit card linked to your Azure account — every resource you create gets billed to a subscription.

💰 What is a Subscription?

A subscription is an agreement with Microsoft to use Azure services. It provides:

  • Billing boundary — each subscription gets its own invoice/bill
  • Access control boundary — you can set different permissions per subscription
  • Resource limits — each subscription has quotas (e.g., max 250 storage accounts)

One Entra ID tenant can have multiple subscriptions:

  • 💻 Dev/Test Subscription — for development and testing (lower costs)
  • 🏭 Production Subscription — for live/production workloads
  • 🧪 Sandbox Subscription — for learning and experimentation

🏠 Real-Life Analogy

Think of subscriptions like bank accounts:

  • You (Entra ID) are the account holder
  • You can have multiple bank accounts (subscriptions)
  • Each account has its own balance and statements (separate billing)
  • You can give different family members different access — your spouse can withdraw, your child can only view the balance
  • Each account has limits — daily withdrawal cap, transaction limits

Keeping Dev and Prod in separate subscriptions is like having a personal and business bank account — you never mix the money.

📁 3. Resource Groups

💡 Simple definition: A Resource Group is a logical container that holds related Azure resources. Think of it as a folder on your computer — you group related files together.

📂 What is a Resource Group?

Every Azure resource (VM, database, storage account) must belong to a Resource Group. Key rules:

  • A resource can belong to only one Resource Group
  • A Resource Group can contain resources from different regions
  • The Resource Group itself has a location (for metadata storage only)
  • You can move resources between Resource Groups
  • Deleting a Resource Group deletes ALL resources inside it ⚠️
  • You can apply tags, policies, and RBAC at the Resource Group level

🏠 Real-Life Analogy

Think of Resource Groups like rooms in a house:

  • Kitchen (RG) → contains stove, fridge, microwave (resources)
  • Bedroom (RG) → contains bed, wardrobe, AC (resources)
  • Office (RG) → contains desk, laptop, monitor (resources)

You group things by purpose. If you demolish the kitchen (delete the RG), everything inside goes — stove, fridge, all of it.

This is why you group related resources together — easy to manage, monitor, and clean up.

❓ Common Interview Questions

Q1: What is the difference between Entra ID and a Subscription?

Entra ID (Tenant) manages identity — who you are, authentication, groups. Subscription manages billing and access — what you can create and who pays for it. One tenant can have many subscriptions.

Q2: Can a Resource Group contain resources from different regions?

Yes! A Resource Group has a location (for metadata), but the resources inside can be in any Azure region. For example, your RG might be in "Central India" but contain a VM in "East US" and a database in "West Europe".

Q3: What happens when you delete a Resource Group?

ALL resources inside it are deleted! This is both a feature (easy cleanup) and a risk. Always double-check before deleting a Resource Group, especially in production. Use resource locks to prevent accidental deletion.

Q4: What is the difference between Authentication and Authorization?

Authentication = verifying who you are (login with username + password). Authorization = determining what you're allowed to do (RBAC roles). Entra ID handles both. First you prove your identity (AuthN), then Azure checks your permissions (AuthZ).

Q5: Can a resource exist without a Resource Group?

No. Every Azure resource MUST belong to exactly one Resource Group. You cannot create a resource without specifying a Resource Group.

🔐 Service Principals, App Registration & Managed Identities

How applications authenticate to Azure — the identity story for non-humans

Teaching hook: "We learned how humans login to Azure — username, password, MFA via Entra ID. But what about applications? Your Python script, your web app, your data pipeline — they can't type a password. So how do they prove their identity? That's what this section is about."

👤 Humans vs. 🤖 Applications — How They Access Azure

👤 How Humans Access Azure

When you (a person) want to use Azure:

  1. Open portal.azure.com in your browser
  2. Enter your email (e.g., ajay@techcorp.com)
  3. Enter your password
  4. Complete MFA (phone notification / authenticator code)
  5. Entra ID verifies you → Access granted!
  6. Your RBAC role determines what you can do (read, create, delete)

✅ This works because you're a human — you can type a password and approve an MFA prompt.

🤖 How Applications Access Azure

When an application (code) needs to talk to Azure:

  1. The app can't open a browser or type a password
  2. It can't do MFA — no phone, no fingerprint
  3. It needs to run 24/7 unattended — no human to click "Approve"
  4. So it needs a different way to prove its identity

Two solutions:

  • 🔑 App Registration (Service Principal) — Client ID + Client Secret
  • 🛡️ Managed Identity — Azure handles everything automatically

⚠️ An application needs an identity in Entra ID, just like a human user does. Without it, Azure won't let it access any resource.

🔑 App Registration (Service Principal / SPN)

💡 Simple definition: An App Registration creates an identity for your application in Entra ID. It gives your app a Client ID (username) and Client Secret (password) so it can authenticate — just like a human uses email + password.

📋 Key Terms (They All Mean the Same Thing!)

You'll hear these terms used interchangeably — they're all related:

App RegistrationThe process of registering your app in Entra ID (what you do in the portal)
Service Principal (SPN)The identity object created in Entra ID when you register the app — it's the "user account" for your app
Application (Client) IDA unique GUID — like the app's username. Never changes.
Client SecretA generated password string — like the app's password. You set an expiry (6 months, 1 year, 2 years).
Tenant IDWhich Entra ID tenant this app belongs to (your organization).
Directory (Tenant) IDSame as Tenant ID — just another name.

Think of it this way: App Registration = creating the account, SPN = the account itself, Client ID = username, Client Secret = password.

Step 5: What Happens Behind the Scenes?

// The authentication flow:

1. App sends Client ID + Client Secret + Tenant ID
   → to https://login.microsoftonline.com/{tenant-id}/oauth2/v2.0/token

2. Entra ID verifies: "Is this Client ID valid? Is the secret correct?"
   → ✅ Yes → Returns an Access Token (JWT, valid for ~1 hour)

3. App sends the Access Token in the HTTP header:
   → Authorization: Bearer eyJ0eXAiOiJKV1Qi...

4. Azure Blob Storage receives the request:
   → Validates the token
   → Checks RBAC: "Does this SPN have Storage Blob Data Contributor role?"
   → ✅ Yes → Upload allowed!

🛡️ Managed Identities — The Better Way!

Teaching hook: "App Registration works, but you have to manage secrets — rotate them, store them securely, worry about expiry. What if Azure could handle all of that automatically? That's exactly what Managed Identities do."

💡 Simple definition: A Managed Identity is an identity automatically created and managed by Azure for your resource. No Client ID, no Client Secret, no secret rotation — Azure handles everything. Your app just says "I am this Azure resource" and gets authenticated.

🔧 System-Assigned Managed Identity

Automatically created with the Azure resource and tied to its lifecycle.

  • Created when: You enable it on a resource (VM, App Service, Data Factory, etc.)
  • Lifecycle: Tied to the resource — if you delete the resource, the identity is also deleted
  • One-to-one: Each resource gets its own unique identity
  • Cannot be shared: Only that specific resource can use this identity

Analogy: Like your office key card — it's issued to you, tied to your employment. If you leave the company, the card is deactivated automatically.

How to enable:

// In Azure Portal:
// Go to your resource (e.g., App Service)
// → Identity → System assigned → Status: ON
// → Save → Azure creates the identity automatically!

// Then assign RBAC role just like before:
// Storage Account → IAM → Add role assignment
// → Storage Blob Data Contributor → Select your App Service

👤 User-Assigned Managed Identity

Created as a standalone Azure resource and can be assigned to multiple resources.

  • Created when: You manually create it as a separate resource
  • Lifecycle: Independent — deleting a VM doesn't delete this identity
  • One-to-many: One identity can be shared across multiple resources
  • Reusable: Assign the same identity to 5 VMs, 3 App Services, etc.

Analogy: Like a shared team access card — the card exists independently. Multiple team members (resources) can use it. If one person leaves, the card still works for others.

How to create:

// In Azure Portal:
// Search "Managed Identities" → Create
Name:            "mi-patient-app"
Resource Group:  "rg-hospital-app"
Region:          "Central India"

// Then assign it to your App Service:
// App Service → Identity → User assigned → Add → Select "mi-patient-app"

// Then assign RBAC role to the managed identity

🤔 When to Use What? — Decision Guide

Use App Registration when...

  • App runs outside Azure (on-prem server, developer laptop, AWS)
  • Third-party app needs to access your Azure resources
  • Multi-tenant scenario (app serves multiple organizations)
  • CI/CD from non-Azure tools (Jenkins, GitLab CI)

Use System-Assigned MI when...

  • App runs on one Azure resource (1 App Service, 1 VM)
  • Identity should be auto-deleted with the resource
  • Simple setup — most common choice
  • E.g., Data Factory accessing Data Lake

Use User-Assigned MI when...

  • Multiple resources need the same identity
  • Identity must survive if a resource is deleted/recreated
  • Pre-configure permissions before creating resources
  • E.g., 5 microservices all accessing the same SQL DB

❓ Interview Questions

Q1: What is a Service Principal in Azure?

A Service Principal (SPN) is an identity for an application in Microsoft Entra ID. It's created via App Registration and allows the app to authenticate using a Client ID and Client Secret (or certificate). Q2: What is the difference between System-Assigned and User-Assigned Managed Identity?

Q3: Why should you prefer Managed Identity over App Registration?

Managed Identities eliminate secrets entirely — no Client Secret to store, rotate, or risk leaking. Azure handles credential management automatically. Q4: What is DefaultAzureCredential in Python?

DefaultAzureCredential from the azure-identity library is a smart credential class that tries multiple auth methods in order: Managed Identity (in Azure) → Environment Variables → Azure CLI → VS Code login. It automatically picks the right one, so the same code works locally (using your CLI login) and in production (using Managed Identity).

Storage Blob Data Contributor — this role allows the app to read, write, and delete blobs in Azure Blob Storage. It was assigned because the app needs to upload patient documents (PDFs, X-rays) to blob containers. 📊 Nature of Data — Structured, Semi-Structured & Unstructured

▼

Before we dive into Azure Storage, we need to understand the different types of data — because the type of data determines which storage solution you use.

Teaching hook: "As a Data Engineer, the first question you ask is not 'where do I store this?' — it's 'what kind of data is this?' The answer determines everything — the storage, the processing tool, the query language, and the cost."

The Three Types of Data

📊 Structured Data

Data that follows a fixed schema — organized in rows and columns like a spreadsheet or database table. Every row has the same columns, same data types, same format.

Think: Excel spreadsheet, SQL database table

📦 Semi-Structured Data

Data that has some organization (tags, keys, hierarchy) but no fixed schema. Each record can have different fields. It's self-describing — the structure is embedded in the data itself.

Think: JSON, XML, CSV with varying columns

🌊 Unstructured Data

Data with no predefined structure at all. Can't be organized in rows and columns. It's the most common type of data — over 80% of all data generated is unstructured.

Think: Images, videos, PDFs, emails, social media posts

📊 1. Structured Data — In Detail

✅ Characteristics

  • Fixed schema — every record has the same columns
  • Stored in tables (rows × columns)
  • Each column has a defined data type (int, varchar, date, etc.)
  • Easy to search, filter, and query using SQL
  • Relationships between tables (foreign keys, joins)

📝 Example: Employee Table

-- SQL Table: Employees
| EmpID | Name       | Dept        | Salary  | JoinDate   |
|-------|------------|-------------|---------|------------|
| 101   | Ajay       | Engineering | 95000   | 2022-03-15 |
| 102   | Priya      | Marketing   | 82000   | 2023-01-10 |
| 103   | Rahul      | Data Team   | 90000   | 2021-07-22 |

-- Every row has the SAME columns
-- Every column has a FIXED data type
-- Easy to query: SELECT * FROM Employees WHERE Dept = 'Data Team'

📦 Real-World Examples of Structured Data

BankingTransaction records — date, amount, account number, type (debit/credit)
E-CommerceOrder history — order ID, product, quantity, price, customer ID
HR SystemsEmployee data — name, department, salary, joining date
HospitalPatient records — patient ID, name, age, diagnosis, doctor

Storage: Azure SQL Database, Azure Synapse Analytics, PostgreSQL, MySQL

📦 2. Semi-Structured Data — In Detail

✅ Characteristics

  • No fixed schema — different records can have different fields
  • Has tags, keys, or markers to organize data (key-value pairs, nesting)
  • Self-describing — the structure is in the data itself
  • Flexible — new fields can be added without changing the schema
  • Common formats: JSON, XML, YAML, Parquet, Avro, CSV

📝 Example: JSON (most common)

    "state": "Telangana"

// Different fields, different nesting — but still organized

Teaching hook: "An Azure Storage Account is like a Swiss Army knife for data — it can store blobs (files), tables (NoSQL), queues (messages), and file shares. It's the most fundamental service in Azure. Almost every project uses it."

🏦 What is an Azure Storage Account?

A Storage Account is a container that groups Azure storage services under a single namespace. It provides a unique namespace accessible from anywhere in the world over HTTP/HTTPS.

📋 Key Properties

  • Globally unique name — 3-24 characters, lowercase letters and numbers only
  • URL format: https://{account-name}.blob.core.windows.net
  • Contains multiple storage services (Blob, Table, Queue, File)
  • Has its own access keys, redundancy settings, and performance tier
  • Lives inside a Resource Group and a region

📦 Four Services Inside a Storage Account

One storage account gives you access to 4 different storage services:

📦 Blob Storage

Stores unstructured data — files of any type (images, videos, PDFs, logs, backups, Parquet files). The most used service.

URL: https://{account}.blob.core.windows.net

Use for: Data lakes, media hosting, backups, log storage, static websites

Similar to: AWS S3, Google Cloud Storage

📁 Azure Files

Fully managed file shares in the cloud, accessible via SMB (Windows) or NFS (Linux) protocol.

URL: https://{account}.file.core.windows.net

Use for: Replace on-prem file servers, shared drives, lift-and-shift of legacy apps

Analogy: Like a shared network drive (\\server\share) but in the cloud

📊 Table Storage

A NoSQL key-value store for structured, non-relational data. Schema-less — each row can have different columns.

URL: https://{account}.table.core.windows.net

Use for: IoT device data, user profiles, configuration data, session state

Note: For serious NoSQL needs, use Azure Cosmos DB instead (Table Storage is basic)

📨 Queue Storage

Store and process messages asynchronously. Each message up to 64 KB. Millions of messages per queue.

URL: https://{account}.queue.core.windows.net

Use for: Decouple application components, background job processing, task queues

Note: For advanced messaging, use Azure Service Bus instead

🌡️ Access Tiers — Hot, Cool, Cold & Archive

Teaching hook: "Not all data is accessed equally. Today's sales data is read 100 times a day. Last year's data is read once a month. 5-year-old compliance data might never be read again. Why pay the same price for all of them?"

🔥 Hot Tier

For data that is accessed frequently.

  • Storage cost: Highest 💰💰💰
  • Access cost: Lowest
  • Use for: Active data — today's logs, current datasets, live application data

❄️ Cool Tier

For data accessed infrequently (at least 30 days old).

  • Storage cost: Lower (~50% less than Hot)
  • Access cost: Higher than Hot
  • Use for: Short-term backups, older datasets, monthly reports
  • Min storage: 30 days (early deletion fee if removed sooner)

🧊 Cold Tier

For data accessed rarely (at least 90 days old).

  • Storage cost: Even lower
  • Access cost: Higher than Cool
  • Use for: Quarterly data, disaster recovery copies
  • Min storage: 90 days

🗄️ Archive Tier

For data that is almost never accessed but must be retained.

  • Storage cost: Cheapest 💰
  • Access cost: Very high + takes hours to rehydrate (data is offline)
  • Use for: Compliance data (7-year retention), legal hold, old backups
  • Min storage: 180 days
  • Rehydration: Standard (up to 15 hours) or High Priority (under 1 hour)

🆚 Blob Storage vs. Azure Data Lake Storage Gen2 (ADLS)

💡 Key insight: ADLS Gen2 is NOT a separate service. It's Blob Storage with Hierarchical Namespace (HNS) enabled. Same underlying infrastructure, but with real folder structure and better performance for analytics.

📦 Blob Storage (without HNS)

  • Flat namespace — "folders" are just naming conventions (virtual)
  • Renaming a "folder" means renaming every single blob inside it
  • No atomic directory operations
  • Good for: General file storage, media, backups, static websites

🏔️ ADLS Gen2 (with HNS) ⭐

  • Hierarchical namespace — real directories and subdirectories
  • Renaming a directory is one atomic operation (instant, regardless of files inside)
  • POSIX-like permissions (ACLs) on directories
  • Optimized for big data analytics (Spark, Databricks, Synapse)
  • Good for: Data lakes, ETL pipelines, analytics workloads

🔄 Redundancy Options — How Azure Protects Your Data

Azure always stores multiple copies of your data to protect against hardware failures. Before we look at the options, let's understand the infrastructure:

🏢 Data Centers, Availability Zones & Regions — What Are They?

🏢 Data Center

A physical building (or set of buildings) filled with thousands of servers, storage devices, networking equipment, cooling systems, and backup power (UPS, generators).

  • Think of it as a giant warehouse of computers
  • Has multiple racks, each rack has multiple servers
  • Has its own power supply, cooling, and network
  • If a disk fails → data is safe (multiple disks). If a rack fails → data is safe (multiple racks)
  • But if the entire building loses power or floods → all data in that DC is at risk

Analogy: One branch office of a bank.

🏗️ Availability Zone (AZ)

A group of one or more data centers within a region, each with independent power, cooling, and networking.

  • Each Azure region typically has 3 Availability Zones (AZ1, AZ2, AZ3)
  • Zones are physically separate — different buildings, sometimes kilometers apart
  • Connected by high-speed, low-latency private fiber
  • If one zone goes down (fire, flood, power outage) → other zones keep running

Analogy: Three different bank branches in the same city — if one branch catches fire, the other two still operate.

🌍 Region

A geographic area containing multiple data centers / Availability Zones. Examples: Central India (Pune), East US (Virginia), West Europe (Netherlands).

  • Azure has 60+ regions worldwide
  • Regions are hundreds/thousands of km apart
  • Each region is paired with another region for disaster recovery (e.g., Central India ↔ South India)
  • If an entire region goes down (earthquake, major disaster) → paired region takes over

Analogy: Bank branches in different cities — if Pune is hit by an earthquake, Mumbai branches still work.

🌐 Public (Anonymous) Access

⚠️ Important security topic! By default, Azure storage is private. But you CAN configure public access — and this is a common misconfiguration that leads to data breaches.

🔒 No Public Access (Default)

All requests must be authenticated. No anonymous access at all.

Use for: All sensitive data — databases, data lakes, application data. This should be your default!

📁 Blob-Level Public Access

Anyone can read a specific blob if they know the URL. They cannot list the container contents.

Use for: Public images (product photos on a website), downloadable files. User must know the exact URL.

📦 Container-Level Public Access

Anyone can list all blobs in the container AND read them. Most permissive — most dangerous.

Use for: Rarely appropriate. Maybe public datasets. Never for sensitive data!

🔧 Two Levels of Control

1. Storage Account levelSetting: "Allow Blob public access" — if disabled, NO container in this account can ever be public. This is the master switch.
2. Container levelIf the master switch is ON, each container can individually be set to: Private, Blob, or Container access.

Best practice: Disable public access at the storage account level. If you need to share files publicly, use SAS tokens with expiry instead.

📸 Blob Versioning & Snapshots

📋 Blob Versioning

When enabled, Azure automatically saves a new version every time a blob is modified or overwritten.

  • Automatic — no extra code needed, Azure does it
  • Each version has a unique version ID (timestamp-based)
  • You can list, read, restore, or delete any previous version
  • Current version is always accessible at the normal URL
  • Previous versions add to storage costs

Use for: Protecting against accidental overwrites, audit trails, maintaining data history

// Version URLs:
// Current:  .../container/file.csv
// Previous: .../container/file.csv?versionid=2024-01-15T10:30:00

📸 Blob Snapshots

A manual, read-only copy of a blob at a specific point in time.

  • Manual — you explicitly create a snapshot (via portal, CLI, or code)
  • Read-only — you can read it but cannot modify a snapshot
  • Stored as deltas (only changes are stored, saving space)
  • Same URL + ?snapshot=datetime parameter
  • Can be promoted back to the current version

Use for: Backups before major changes, point-in-time recovery, data comparison

// Snapshot URL:
.../container/file.csv?snapshot=2024-01-15T10:30:00.0000000Z

🗑️ Soft Delete

When enabled, deleted blobs are not immediately removed. They're moved to a "soft deleted" state and can be recovered within the retention period (1–365 days).

  • Blob soft delete — recover deleted blobs
  • Container soft delete — recover deleted containers (and all blobs inside)
  • Works with versioning — soft-deleted versions can also be recovered

Best practice: Always enable soft delete (7-14 days retention) as a safety net against accidental deletion.

  • Distributed locks — ensure only one process writes to a file at a time
  • Prevent accidental deletion — lease a blob so nobody can delete it
  • ETL pipelines — lock a file while processing to prevent concurrent writes

🔐 Authentication Methods — How to Access Storage

There are multiple ways to authenticate to Azure Storage. Each has different security levels and use cases:

🔑 1. Access Keys (Storage Account Keys)

Two 512-bit keys (key1, key2) generated when the storage account is created. Provide full access to everything in the account.

  • Scope: Entire storage account — all containers, all blobs, all services
  • Security: ⚠️ Most dangerous — like a master key. If leaked, attacker has full access.
  • Two keys so you can rotate one while using the other (zero downtime rotation)
  • Never share in code, emails, or repos. Store in Key Vault.

Analogy: The master key to the entire building. Opens every door. Very dangerous if lost.

🎫 2. Shared Access Signature (SAS)

A signed URL that grants limited, time-bound access to specific resources.

What you can control:

  • Scope: Which container, which blob, which operations (read/write/delete)
  • Time-bound: Set start and expiry time
  • IP restriction: Limit to specific IP addresses
  • Protocol: Can enforce HTTPS only

Three types of SAS:

  • Account SAS — access to one or more services in the account
  • Service SAS — access to a single service (e.g., Blob only)
  • User Delegation SAS — signed with Entra ID credentials (most secure) ⭐

Analogy: A visitor pass — valid for specific floors, specific hours, specific date. Expires automatically.

When generating an Account SAS in the Azure Portal, you're asked to select Allowed Resource Types. These control what level of resources the SAS token can access:

ServiceAccess to service-level operations — things that affect the service itself, not individual files.List containers, get/set service properties, get service stats

Service = the building manager (manages the building itself)
Container = the floor manager (manages rooms/floors)
Object = the employee (works on actual files/documents)

  • Define: start time, expiry time, permissions

📜 Immutable Storage & Legal Hold

🔒 Immutability Policies (WORM)

Write Once, Read Many — once data is written, it cannot be modified or deleted for a specified period.

  • Time-based: Data is immutable for X days (e.g., 365 days, 7 years)

  • No expiry — must be manually removed
  • Multiple holds can be applied (tagged with hold IDs)
  • Data is protected even if the time-based policy expires

Analogy: A court order saying "do not destroy these documents until the case is resolved."

📊 Azure Table Storage — Deep Dive

💡 Simple definition: Azure Table Storage is a NoSQL key-value store for massive amounts of structured, non-relational data. It's schema-less — each row (entity) can have different columns. Think of it as a giant, flexible spreadsheet with billions of rows.

📋 Key Concepts

  • Table — A collection of entities (like a database table, but no schema)
  • Entity — A single row in the table (like a record/row)
  • Property — A column/field on an entity (key-value pair)
  • Each entity can have up to 252 properties
  • Max entity size: 1 MB
  • No joins, no foreign keys, no complex queries — it's simple key-value lookup

🏗️ Table Hierarchy

Storage Account
  └── Table (e.g., "Customers")
        └── Entity (one row)
              ├── PartitionKey (required)
              ├── RowKey (required)
              ├── Timestamp (auto-generated)
              └── Custom properties (flexible)

URL: https://{account}.table.core.windows.net/{table}

🔑 PartitionKey & RowKey — The Most Important Concept!

Teaching hook: "Every entity in Table Storage MUST have two keys — PartitionKey and RowKey. Together, they form the unique identifier (primary key) for that entity. Understanding these is crucial for performance."

🗂️ PartitionKey

Determines which partition (server) the entity is stored on. All entities with the same PartitionKey are stored together on the same node.

  • Purpose: Grouping — entities with the same key are co-located
  • Performance: Queries filtering on PartitionKey are fast (single node lookup)
  • Scalability: Different partition keys are automatically spread across servers
  • Type: String value

Analogy: Like a filing cabinet drawer — all documents for "Sales" go in the Sales drawer. To find a Sales document, you go directly to that drawer.

🔍 RowKey

A unique identifier within the partition. The combination of PartitionKey + RowKey is unique across the entire table.

  • Purpose: Unique ID within a partition group
  • Performance: Point queries (PartitionKey + RowKey) are the fastest possible query
  • Sorting: Entities within a partition are sorted by RowKey
  • Type: String value

Analogy: Like the file name/label inside the drawer — once you're in the "Sales" drawer, the RowKey tells you exactly which document to grab.

🧑‍🏫 Example: IoT Sensor Data

// Table: SensorReadings
// PartitionKey = DeviceId (group by device)
// RowKey = Timestamp (unique reading per device)

| PartitionKey | RowKey                | Temperature | Humidity | Location    |
|--------------|-----------------------|-------------|----------|-------------|
| sensor-001   | 2024-01-15T10:30:00  | 23.5        | 65       | Warehouse-A |
| sensor-001   | 2024-01-15T10:35:00  | 24.1        | 63       | Warehouse-A |
| sensor-001   | 2024-01-15T10:40:00  | 23.8        | 64       | Warehouse-A |
| sensor-002   | 2024-01-15T10:30:00  | 18.2        | 70       | Cold-Room   |
| sensor-002   | 2024-01-15T10:35:00  | 18.0        | 71       | Cold-Room   |

// Query all readings for sensor-001 → fast! (same partition)
// Query sensor-001 at 10:30 → fastest! (PartitionKey + RowKey = point query)
// Query all sensors at 10:30 → slow! (cross-partition scan)

📏 PartitionKey Design — Best Practices

Scenario Good PartitionKey Good RowKey Why
IoT sensor dataDeviceIdTimestampQuery all readings for one device (same partition)

Log dataDate (YYYY-MM-DD)LogId / TimestampQuery all logs for a specific day
User sessionsUserIdSessionIdGet all sessions for one user

⚠️ PartitionKey design is the most important decision in Table Storage! Get it wrong, and your table will either crash under load or be painfully slow to query. Let's look at the two extremes:

❌ Bad: Same PartitionKey for Everything ("Hot Partition")

// All 10 million entities have PartitionKey = "all"
| PartitionKey | RowKey   | Name   | City      |
|--------------|----------|--------|-----------|
| all          | 001      | Ajay   | Hyderabad |
| all          | 002      | Priya  | Mumbai    |
| all          | 003      | Rahul  | Delhi     |
| all          | ...      | ...    | ...       |
| all          | 10000000 | Sneha  | Chennai   |

What happens:

  • ALL 10 million entities sit on a single partition server
  • That one server handles every single request — reads, writes, everything
  • The server gets overwhelmed → requests start failing or timing out
  • Other partition servers sit idle — you paid for scale but aren't using it
  • This is called a "hot partition" — one server is burning while others are cold

Analogy: Everyone in a mall going to one cashier even though 10 cashiers are available. Huge queue at one counter, others empty.

❌ Bad: Unique PartitionKey for Every Entity (Too Scattered)

// Every entity has a unique GUID as PartitionKey
| PartitionKey                         | RowKey | Name   |
|--------------------------------------|--------|--------|
| a1b2c3d4-...-1111                   | 001    | Ajay   |
| e5f6g7h8-...-2222                   | 001    | Priya  |
| i9j0k1l2-...-3333                   | 001    | Rahul  |

What happens:

  • Every entity is on a different partition server
  • Point queries (one entity) are fine
  • But range queries are impossible — "get all customers in Mumbai" hits every partition (full table scan)
  • No grouping — related data is scattered across servers
  • Batch operations fail — batch requires entities in the same partition

Analogy: Every book in a library on a different floor. Finding one book is fine, but finding all "Science" books means searching every floor.

✅ Good: Balanced PartitionKey (Group Related Data)

// PartitionKey = City (groups related customers together)
| PartitionKey | RowKey | Name   | Age | Email             |
|--------------|--------|--------|-----|-------------------|
| Hyderabad    | 001    | Ajay   | 28  | ajay@tc.com       |
| Hyderabad    | 002    | Sneha  | 25  | sneha@tc.com      |
| Hyderabad    | 003    | Kiran  | 30  | kiran@tc.com      |
| Mumbai       | 001    | Priya  | 26  | priya@tc.com      |
| Mumbai       | 002    | Rohan  | 29  | rohan@tc.com      |
| Delhi        | 001    | Rahul  | 27  | rahul@tc.com      |

Why this works:

  • Related data is grouped — all Hyderabad customers on one partition, Mumbai on another
  • "Get all customers in Hyderabad" → fast! Single partition query
  • "Get Ajay in Hyderabad" (PartitionKey + RowKey) → fastest! Point query
  • Data is distributed across multiple servers — no hot partition
  • Batch operations work — batch all Hyderabad updates together

Analogy: A library organized by genre — all Science books on one shelf, all History on another. Finding all Science books is easy, and no single shelf is overloaded.

📏 The Golden Rule of PartitionKey Design

Question to AskAnswer Guides Your Design
"How will I query this data most often?"The most common query filter = your PartitionKey
"What data do I need together?"Data queried together should share a PartitionKey
"How many entities per partition?"Aim for thousands to low millions — not 1 and not billions
"Will one partition get way more traffic?"If yes → that's a hot partition. Choose a more even distribution.

📨 Azure Queue Storage — Deep Dive

💡 Simple definition: Azure Queue Storage is a service for storing large numbers of messages that can be processed asynchronously. It decouples application components — the sender and receiver don't need to be available at the same time.

📋 Key Concepts

  • Queue — A container for messages
  • Message — A piece of data (text or JSON), max 64 KB
  • FIFO-ish — generally first-in-first-out, but not strictly guaranteed
  • A queue can hold millions of messages
  • Messages can have a visibility timeout — hidden from other consumers while being processed
  • Messages can have a TTL (Time to Live) — auto-deleted after expiry (default: 7 days)
  • Max queue size: 500 TiB

🔄 How It Works

// The decoupling pattern:

Producer (Web App)
  │
  │  "New order: #1234, ₹2,500"
  ▼
📨 Queue (messages waiting)
  │  ┌──────────────────────┐
  │  │ Order #1234 - ₹2,500 │
  │  │ Order #1235 - ₹800   │
  │  │ Order #1236 - ₹3,200 │
  │  └──────────────────────┘
  │
  ▼
Consumer (Background Worker)
  │
  └── Processes orders one by one
      Sends confirmation email
      Updates inventory

🧑‍🏫 Real-World Example: E-Commerce Order Processing

Without queue: When a customer places an order, the web app must immediately: validate payment → update inventory → send confirmation email → generate invoice. If any step fails or is slow, the customer waits or gets an error.

With queue: The web app just drops a message in the queue ("Order #1234 placed") and immediately responds to the customer with "Order received!" A background worker picks up messages and processes them independently — even if it takes 30 seconds, the customer doesn't wait.

Without QueueWith Queue
Customer waits for all steps to completeCustomer gets instant response
If email service is down → order failsIf email is down → message stays in queue, retried later
Web app must handle everythingWeb app and worker scale independently
Coupled — all components must be onlineDecoupled — components work independently

📨 Message Lifecycle

1. Producer adds message to queue → message is visible

2. Consumer reads message → message becomes invisible
   (visibility timeout starts, default 30 seconds)

3a. Consumer processes successfully → deletes the message ✅

3b. Consumer crashes / timeout expires → message becomes
   visible again → another consumer picks it up (auto-retry!) 🔄

// This ensures no message is ever lost — even if a worker crashes

🌐 What is REST?

💡 REST (REpresentational State Transfer) is an architectural style for designing APIs. It uses standard HTTP methods, URLs, and status codes. A REST API is just an API that follows REST rules.

📏 REST Principles

1. Client-ServerClient (your app) and Server (API) are separate. Client makes requests, server sends responses.
2. StatelessEach request is independent — the server doesn't remember previous requests. Every request must include all needed info (auth token, parameters).
3. Uniform InterfaceUse standard HTTP methods (GET, POST, PUT, DELETE) and URLs to interact with resources.
4. Resource-BasedEverything is a resource identified by a URL. Example: /api/customers/101 = customer with ID 101.
5. JSON ResponseData is typically exchanged in JSON format (lightweight, human-readable).

🔐 Bearer Token — How APIs Authenticate

💡 A Bearer Token is an access credential — whoever "bears" (carries) the token gets access. It's like a movie ticket — the theater doesn't care who bought it, whoever shows it gets in.

🔄 How Bearer Token Authentication Works

1. Your App sends credentials to Entra ID
   (Client ID + Client Secret, or Managed Identity)

2. Entra ID validates credentials
   → Returns an Access Token (JWT)
   → Token is valid for ~1 hour

3. Your App includes the token in every API call:
   Authorization: Bearer eyJhbGciOi...

4. Azure Service (e.g., Blob Storage) receives the call:
   → Validates the token (is it expired? is it real?)
   → Checks permissions (does this identity have the right role?)
   → ✅ Returns data / ❌ Returns 401 or 403

📋 What's Inside a JWT Token?

A JWT (JSON Web Token) has 3 parts separated by dots:

eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiIxMjM0NSJ9.SflKxwRJSMeKKF2QT4fw
└── Header ──┘  └── Payload ──┘  └── Signature ──┘
HeaderAlgorithm used (RS256), token type (JWT)
PayloadClaims — who you are, what you can do, when it expires:
sub (subject/user), aud (audience/API), exp (expiry), iss (issuer), roles
SignatureCryptographic signature — proves the token wasn't tampered with

You can decode any JWT at jwt.ms (Microsoft) or jwt.io — paste the token and see its contents.

🐍 Python Example — Calling Azure REST API with Bearer Token

import requests
from azure.identity import DefaultAzureCredential

# Step 1: Get the Bearer Token
credential = DefaultAzureCredential()
token = credential.get_token("https://storage.azure.com/.default")

# Step 2: Call Azure Blob Storage REST API
url = "https://stcretail.blob.core.windows.net/raw-data?restype=container&comp=list"

headers = {
    "Authorization": f"Bearer {token.token}",    # Bearer token in header
    "x-ms-version": "2023-11-03",               # Azure API version
    "x-ms-date": "Sat, 15 Jan 2024 10:30:00 GMT"
}

response = requests.get(url, headers=headers)

print(f"Status: {response.status_code}")   # 200 = success
print(f"Body: {response.text}")              # XML list of blobs

Section A: Cloud Computing Fundamentals — Answers

1. What is cloud computing?

Cloud computing is the delivery of computing services (servers, storage, databases, networking, software, analytics, AI) over the internet ("the cloud") on a pay-as-you-go basis. Instead of buying and maintaining your own physical servers, you rent them from a cloud provider like Azure, AWS, or GCP. Example: Instead of buying a server for ₹10 lakhs to host your website, you use Azure App Service and pay ₹2,000/month. If traffic increases during a sale, Azure automatically adds more capacity.

2. List and explain the 5 essential characteristics of cloud computing (NIST).

1) On-Demand Self-Service — Get resources instantly without human intervention (like ordering on Swiggy). 2) Broad Network Access — Access from anywhere on any device (like Netflix). 3) Resource Pooling — Provider's resources are shared across customers using multi-tenant model (like an apartment building). 4) Rapid Elasticity — Scale up/down instantly based on demand (like a rubber band). 5) Measured Service — Pay only for what you use with precise metering (like an electricity meter).

3. What is the difference between CapEx and OpEx?

CapEx (Capital Expenditure) — Large upfront spending on physical assets you own (buying servers, data centers). Value depreciates over time. Like buying a car. OpEx (Operational Expenditure) — Ongoing spending on services as you use them (cloud subscriptions). No ownership, no depreciation. Like using Uber. Cloud computing follows OpEx because you pay monthly for what you use with no upfront hardware purchase.

4. Explain the Pay-As-You-Go model with a real-life analogy.

Pay-As-You-Go means you only pay for the resources you actually consume, just like an electricity bill — you pay for the units consumed, not a flat fee. Used a VM for 3 hours? Pay for 3 hours. Stored 50 GB? Pay for 50 GB. Didn't use anything? Pay ₹0. No upfront cost, no long-term commitment, stop anytime.

5. Compare On-Premises vs. Cloud Computing (4 differences).

1) Cost: On-prem = high upfront CapEx; Cloud = low monthly OpEx. 2) Scaling: On-prem = weeks (buy hardware); Cloud = minutes (click a button). 3) Maintenance: On-prem = you manage everything; Cloud = provider manages infra. 4) Disaster Recovery: On-prem = expensive second data center; Cloud = built-in replication across regions.

6. List 3 advantages and 3 disadvantages of cloud computing.

Advantages: 1) Cost efficiency (pay-as-you-go, no hardware), 2) Scalability (scale up/down in minutes), 3) Global reach (deploy to 60+ regions worldwide). Disadvantages: 1) Internet dependency (no internet = no access), 2) Vendor lock-in (hard to switch providers), 3) Ongoing costs can surprise you ("cloud bill shock" from forgotten resources).

7. When should a company choose cloud vs. on-premises?

Cloud: A startup launching a food delivery app with unpredictable traffic — cloud provides auto-scaling, zero upfront cost, and global reach. On-Premises: A military defense system with classified data — requires strict physical security, data cannot leave the premises, full control needed.

8. What is the Shared Responsibility Model?

Security and management responsibilities are shared between the cloud provider and the customer. IaaS: Provider manages physical infra (servers, networking, storage, virtualization); you manage OS, runtime, apps, data. PaaS: Provider also manages OS, runtime, middleware; you manage apps and data. SaaS: Provider manages everything except your data and access controls. Key rule: Data is ALWAYS your responsibility regardless of service model.

9. Should a food delivery startup use cloud or on-premises?

Cloud — because: 1) Zero upfront cost (startups have limited budget), 2) Auto-scaling handles unpredictable traffic (lunch/dinner peaks vs. quiet hours), 3) Global deployment to serve customers in different cities, 4) Pay-as-you-go so they only pay for actual usage, 5) Built-in services for notifications, databases, and analytics without building from scratch.

10. Difference between scalability and elasticity?

Scalability = the ability to handle increased load by adding resources (scaling up = bigger machine, scaling out = more machines). Elasticity = the ability to automatically scale up during demand spikes AND scale down when demand drops. Elasticity is scalability + automatic + both directions. Cloud provides elasticity; on-prem provides scalability (manually).

Section D: Azure Hierarchy — Answers

1. Draw and explain the Azure organizational hierarchy.

Top to bottom: 1) Microsoft Entra ID (Tenant) — identity and access management, "who are you?". 2) Management Groups (optional) — organize multiple subscriptions. 3) Subscriptions — billing and access boundary, "who pays?". 4) Resource Groups — logical containers grouping related resources. 5) Resources — actual services (VMs, databases, storage accounts). Permissions inherit downward from top to bottom.

2. What is Microsoft Entra ID? Authentication vs Authorization?

Entra ID (formerly Azure AD) is Azure's cloud-based identity and access management service. Authentication (AuthN): Verifying WHO you are — login with username + password + MFA. Authorization (AuthZ): Determining WHAT you're allowed to do — RBAC roles (Owner, Contributor, Reader). First you authenticate, then Azure checks your authorization.

3. Tenant ID, Primary Domain, and what happens when adding a user?

Tenant ID: A unique GUID assigned to your Entra ID tenant (e.g., a1b2c3d4-...). Never changes. Primary Domain: The default domain you get: yourcompany.onmicrosoft.com. You can also add a custom domain (e.g., techcorp.com). Adding a user: When you create a user named "ajay", the domain automatically attaches — it becomes ajay@techcorp.onmicrosoft.com (or ajay@techcorp.com with custom domain). This is the UPN (User Principal Name).

4. Explain SSO and MFA with examples.

SSO (Single Sign-On): Login once with your Entra ID credentials and access all connected apps without logging in again — Azure Portal, Teams, Outlook, Power BI, DevOps. Like one key opening all doors in your office. MFA (Multi-Factor Authentication): Requires 2+ verification factors — something you know (password) + something you have (phone/authenticator app). Like a bank locker needing both your key and the manager's key.

5. What is an Azure Subscription? Why have multiple?

A Subscription is a billing and access boundary — an agreement with Microsoft to use Azure services. Each subscription gets its own invoice. Why multiple: 1) Separate billing for Dev vs Prod (track costs separately). 2) Different access controls (devs access Dev subscription, only ops access Prod). 3) Resource limits — each subscription has quotas. 4) Organizational separation (different departments).

6. List 4 types of Azure subscriptions.

1) Free Account: $200 credit for 30 days + 55+ free services for 12 months. 2) Pay-As-You-Go: Pay monthly for usage, no commitment. 3) Enterprise Agreement (EA): Custom pricing for large organizations (500+ users), volume discounts. 4) Azure for Students: $100 credit, no credit card required, .edu email needed.

7. What is a Resource Group? Can it contain resources from different regions?

A Resource Group is a logical container that holds related Azure resources (like a folder on your computer). Every resource must belong to exactly one RG. Yes, a Resource Group can contain resources from different regions. The RG itself has a location (for metadata only), but resources inside can be in any Azure region (e.g., RG in Central India containing a VM in East US and a database in West Europe).

8. What happens when you delete a Resource Group? Prevention?

Deleting a Resource Group deletes ALL resources inside it — VMs, databases, storage, everything. This is by design for easy cleanup, but risky for production. Prevention: Use Resource Locks — apply a "Delete" lock or "ReadOnly" lock on critical RGs/resources to prevent accidental deletion, even by Owners.

9. What are Azure Tags? Give 3 examples.

Tags are key-value pairs attached to resources for organization and cost tracking. Examples: 1) Environment: Production (identify prod vs dev resources). 2) CostCenter: CC-1234 (track costs by department). 3) Owner: ajay@techcorp.com (know who created/manages the resource). Importance: Filter billing by tag, automate actions (shutdown Dev VMs at 7 PM), enforce governance ("every resource must have an Owner tag").

10. What is RBAC? Explain Owner, Contributor, Reader. How does inheritance work?

RBAC (Role-Based Access Control) assigns fine-grained permissions. Owner: Full access + can assign roles. Contributor: Create/manage resources but can't assign roles. Reader: View-only access. Inheritance: Permissions flow downward — a role assigned at Subscription level applies to ALL Resource Groups and Resources inside it. At Management Group level, it applies to ALL subscriptions below. Always follow least privilege principle.

Section E: Service Principals & Managed Identities — Answers

1. How do applications authenticate to Azure?

Applications can't use username + password + MFA like humans (they can't type or approve prompts). Instead, they use: 1) App Registration (SPN) — Client ID (username) + Client Secret (password) sent to Entra ID to get an Access Token. 2) Managed Identity — Azure automatically handles authentication with no secrets needed. Both methods result in an Access Token that the app uses to call Azure services.

2. What is App Registration? Explain Client ID, Client Secret, Tenant ID.

App Registration creates an identity for your app in Entra ID (the process). It creates a Service Principal (SPN) (the identity object). Client ID (Application ID): A unique GUID — the app's username. Never changes. Client Secret: A generated password string — has an expiry date, must be stored securely in Key Vault. Tenant ID: Which Entra ID tenant (organization) the app belongs to.

3. Step-by-step: Patient Registration Tool uploading to Blob Storage via App Registration.

Step 1: Create App Registration in Entra ID (name: PatientRegistrationApp) → get Client ID + Tenant ID. Step 2: Generate Client Secret (Certificates & secrets → New client secret, set 12-month expiry) → copy secret value immediately. Step 3: Go to Storage Account → IAM → Add role assignment → assign "Storage Blob Data Contributor" to PatientRegistrationApp. Step 4: In app code, use ClientSecretCredential(tenant_id, client_id, client_secret) to authenticate, then use BlobServiceClient to upload files. Step 5: App sends credentials to Entra ID → gets Access Token → sends token with upload request to Blob Storage.

4. What is a Managed Identity? Why more secure than App Registration?

A Managed Identity is an identity automatically created and managed by Azure for your resource. More secure because: 1) No Client Secret exists — nothing to leak, steal, or expire. 2) No secret rotation needed — Azure handles credentials internally. 3) No secrets in code or config files. 4) Credentials never leave Azure's infrastructure. App Registration requires managing secrets (expiry, rotation, secure storage), which is a security risk.

5. System-Assigned vs User-Assigned Managed Identity.

System-Assigned: Created automatically when enabled on a resource, tied to resource lifecycle (deleted with resource), one-to-one (can't be shared), simplest to set up. User-Assigned: Created as a standalone Azure resource, independent lifecycle (survives resource deletion), one-to-many (shared across resources). Use System-Assigned for single-resource access (e.g., one App Service → Blob). Use User-Assigned when multiple resources need the same identity (e.g., 5 microservices → same SQL DB).

6. Patient Registration Tool with Managed Identity — what changes?

Instead of ClientSecretCredential(tenant_id, client_id, client_secret), use DefaultAzureCredential() — which automatically detects and uses the Managed Identity. No Client ID, no Client Secret, no Tenant ID in the code. Just: 1) Enable System-Assigned MI on the App Service. 2) Assign Storage Blob Data Contributor to the App Service. 3) Use DefaultAzureCredential() in code. Three values removed from code = three fewer things that can leak.

7. When to use App Registration vs Managed Identity? (2 scenarios each)

App Registration: 1) A Python script running on an on-premises server needs to upload data to Azure Data Lake — Managed Identity only works on Azure resources. 2) A third-party vendor's application needs to access your Azure SQL Database. Managed Identity: 1) An Azure Data Factory pipeline needs to read data from Azure Data Lake Storage — enable System-Assigned MI on ADF. 2) Five Azure App Services (microservices) all need to access the same Azure SQL Database — create one User-Assigned MI and assign it to all five.

Section F: Nature of Data — Answers

1. Three types of data with examples.

Structured: Fixed schema in rows and columns (SQL tables, Excel). Example: Employee table with EmpID, Name, Dept, Salary columns. Semi-Structured: Flexible schema with tags/keys (JSON, XML, Parquet). Example: API response in JSON where each record may have different fields. Unstructured: No schema or format. Example: Images, videos, PDFs, emails — makes up 80%+ of all data.

2. Classify hospital data types.

Patient registration forms (name, age, blood group) = Structured (fixed columns, stored in SQL table). Lab reports in JSON format = Semi-Structured (key-value pairs, flexible schema). X-ray images = Unstructured (binary image files, no tabular format).

3. Parquet vs CSV.

CSV: Row-based, human-readable, no data types (everything is text), no compression, slow for large analytics queries. Parquet: Columnar format, binary (not human-readable), supports data types, highly compressed (70-90% smaller), extremely fast for analytics because queries only read needed columns. Parquet is preferred for data lakes because it reduces storage cost and dramatically improves query performance in Spark/Databricks/Synapse.

4. Unstructured data percentage and examples.

Over 80% of all data generated worldwide is unstructured. Examples: 1) Images and photos (product photos, medical X-rays), 2) Videos (CCTV footage, training recordings), 3) Audio (call recordings, podcasts), 4) Documents (PDFs, Word files, scanned papers).

5. Azure storage for each data type.

Structured: Azure SQL Database, Azure Synapse Analytics (SQL pools). Semi-Structured: Azure Cosmos DB (NoSQL), Azure Data Lake Storage (Parquet/JSON files), Azure Blob Storage. Unstructured: Azure Blob Storage (images, videos, PDFs), Azure Data Lake Storage Gen2.

Four services: 1) Blob Storage (unstructured files), 2) Azure Files (SMB/NFS file shares), 3) Table Storage (NoSQL key-value), 4) Queue Storage (asynchronous messages).

2. Naming rules for Storage Accounts.

Must be globally unique across all of Azure, 3-24 characters, lowercase letters and numbers only. No hyphens, underscores, or uppercase. Valid: stcretailproddata, dlsprojectx2024. Invalid: stc-retail-prod (hyphens), StcRetailProd (uppercase).

3. Blob Storage hierarchy and URL structure.

Storage Account (top level) → Container (like a folder, groups related blobs) → Blob (the actual file). URL: https://{account-name}.blob.core.windows.net/{container}/{blob-name}. Virtual folders are created using / in blob names: /raw-data/2024/01/sales.parquet — but these aren't real folders, just naming conventions (unless HNS is enabled).

4. Three types of blobs.

Block Blob: Made of blocks, most common (99% of use cases) — files, images, data lake files. Max ~190 TB. Append Blob: Optimized for append-only operations — log files, audit trails. Max ~195 GB. Page Blob: Optimized for random read/write in 512-byte pages — used internally for Azure VM disks (VHD files). Max 8 TB.

5. Four access tiers.

Hot: Highest storage cost, lowest access cost, no min retention — for frequently accessed data. Cool: ~50% less storage cost, higher access cost, 30-day min retention — for infrequent access. Cold: ~70% less storage, even higher access cost, 90-day min — for rarely accessed data. Archive: ~90% less storage, very high access cost, 180-day min, data is offline (takes hours to rehydrate) — for compliance/archival data.

6. Lifecycle Management Policy example.

A Lifecycle Management Policy automates moving blobs between tiers based on age. Example rule: Move blobs to Cool tier after 30 days of no access → Move to Cold tier after 90 days → Move to Archive tier after 365 days → Delete after 2555 days (7 years). Configured in Storage Account → Lifecycle management.

7. ADLS Gen2 vs Blob Storage.

ADLS Gen2 is NOT a separate service — it's Blob Storage with Hierarchical Namespace (HNS) enabled. HNS gives real directory structure (not virtual folders), atomic directory operations (rename a folder instantly regardless of files inside), POSIX ACLs for fine-grained directory permissions, and optimized performance for analytics workloads (Spark, Databricks). Same cost as regular Blob Storage.

8. Can you enable HNS later?

No. HNS (Hierarchical Namespace) must be enabled at storage account creation time. You cannot enable or disable it after the account is created. This is critical — for data engineering projects, always create the storage account with HNS enabled (ADLS Gen2). If you forget, you must create a new account and migrate data.

9. Four redundancy options.

LRS (Locally Redundant): 3 copies in a single data center — protects against disk/rack failure. Cheapest. ZRS (Zone Redundant): 3 copies across 3 availability zones in the same region — protects against data center failure. GRS (Geo-Redundant): 6 copies — 3 in primary region + 3 in secondary region — protects against entire region failure. RA-GRS (Read-Access Geo): Same as GRS but secondary region is readable — highest availability.

10. Anonymous/public access.

Three levels: 1) No public access (default) — all requests must be authenticated. 2) Blob-level — anyone can read a specific blob by URL but can't list containers. 3) Container-level — anyone can list and read all blobs. Security risk: Misconfigured public access is a leading cause of cloud data breaches. Best practice: Disable at the storage account level (master switch). Use SAS tokens for controlled sharing instead.

11. Versioning vs Snapshots.

Versioning: Automatic — Azure saves a new version on every write/overwrite. Full copy per version. Best for continuous protection. Snapshots: Manual — you explicitly create a read-only copy at a point in time. Stored as deltas (only changes). Best for point-in-time backups before major changes. Both help protect against accidental overwrites or deletions.

12. Blob Lease.

A lease is an exclusive lock on a blob that prevents others from modifying or deleting it. Duration: 15-60 seconds or infinite. Only the lease holder (identified by a Lease ID GUID) can write/delete. Others can still read. Use cases: Distributed locks (one process at a time), prevent accidental deletion, ETL pipelines locking files during processing.

13. Soft Delete.

When enabled, deleted blobs are not immediately removed — they move to a "soft deleted" state and can be recovered within the retention period (1-365 days). Applies to blobs and containers. Always enable with 7-14 days retention as a safety net against accidental deletion. Soft-deleted data incurs storage costs until it expires.

14. Four authentication methods compared.

Access Keys: Full account access, most dangerous, like a master key. Use only for legacy/testing. SAS Tokens: Granular, time-bound access. Good for external sharing. Medium security. Entra ID + RBAC: Role-based, auditable, revocable. High security. Use for all internal apps. Managed Identity: Same as RBAC but no secrets at all — Azure handles credentials automatically. Highest security. Use for all Azure-hosted apps. Priority: Managed Identity > RBAC > SAS > Access Keys.

15. SAS URL components and Stored Access Policy.

SAS (Shared Access Signature) is a signed URL granting limited, time-bound access. Components: sv (service version), st (start time), se (expiry time), sr (resource type: blob/container), sp (permissions: read/write/delete), sig (cryptographic signature). A Stored Access Policy is a server-side policy on a container that SAS tokens can reference. Key advantage: you can revoke a SAS token by deleting/modifying the policy — regular SAS tokens cannot be revoked before expiry.

Section H: Azure Storage — Hands-On Practice — Answers

1. Create a Storage Account with ADLS Gen2.

Steps: Azure Portal → Create a resource → Storage account. Name: st[yourname]practice (lowercase, numbers only). Region: Central India. Performance: Standard. Redundancy: LRS. Under Advanced tab, enable "Hierarchical namespace" (this makes it ADLS Gen2). Click Review + Create. Key learning: HNS cannot be changed after creation — you must decide at creation time.

2. Create containers with different access levels.

Navigate to your storage account → Containers → + Container. Create "raw-data" (Private), "processed-data" (Private), "images" (Blob — anonymous read). Why: raw-data and processed-data contain sensitive analytics data — must be private. Images container holds product photos that need to be publicly accessible via URL on a website, so Blob-level access allows anyone to read a specific image by URL without listing all images.

3. Upload, overwrite, and check versions.

First upload creates the initial blob. Second upload (before versioning) overwrites it — no history kept. Enable versioning: Storage account → Data protection → Turn on versioning. Third upload creates a new version while keeping the previous one. You should see 2 versions (the current + one previous). The version before versioning was enabled is lost — versioning only tracks changes from the point it's enabled.

4. Create Lifecycle Management Policy.

Storage account → Lifecycle management → Add a rule. Rule name: "auto-tier-raw-data". Apply to: Block blobs. Filter: Container = raw-data. Actions: Move to Cool after 30 days since last modification → Move to Archive after 180 days → Delete after 365 days. This automates cost optimization — frequently accessed data stays Hot, old data moves to cheaper tiers, and very old data is cleaned up.

5. Generate and test SAS token.

Storage account → Shared access signature. Select: Allowed services = Blob. Resource types = Container + Object. Permissions = Read + List. Start/Expiry = now to 1 hour. Protocol = HTTPS only. Click Generate. Copy the Blob service SAS URL. Paste in incognito browser — you should see the container contents (list blobs). After expiry, the URL returns 403 Forbidden. Key learning: SAS tokens are time-bound and cannot be revoked unless using a Stored Access Policy.

6. Access Keys and rotation.

Storage account → Access keys → Show keys → Copy Key1. Use it to connect in Storage Explorer. Then click "Rotate" on Key1 — Azure generates a new Key1. The old Key1 immediately stops working — any app using it will break. Two keys exist so you can: 1) Switch all apps to Key2, 2) Rotate Key1, 3) Switch apps to new Key1, 4) Rotate Key2. This enables zero-downtime key rotation.

7. Create Azure Table with PartitionKey/RowKey.

Storage account → Tables → + Table → Name: SensorData. Use Storage Explorer or Azure Portal to add entities. Each entity must have PartitionKey and RowKey (strings). Add 5 entities with PK="sensor-001" and RK as timestamps, plus custom properties (Temperature, Humidity, Location). Add 3 more with PK="sensor-002". Query: filter PartitionKey eq 'sensor-001' — returns only the 5 sensor-001 entities (fast, single partition). Key learning: PartitionKey groups related data for fast queries.

8. Create File Share and mount.

Storage account → File shares → + File share → Name: team-docs, Quota: 1 GiB. Upload 3 files. Click "Connect" → select your OS (Windows) → copy the PowerShell script → run in terminal. It mounts as a network drive (e.g., Z:). Protocol: SMB (Server Message Block), port 445. Note: Some ISPs/corporate firewalls block port 445 — if mounting fails from home, it's likely a port block.

9. Create Queue and test messages.

Storage account → Queues → + Queue → Name: order-queue. Add 3 messages. Peek: Views messages without removing them — the message stays in the queue for other consumers. Dequeue: Removes the message from the queue (marks it invisible, then you delete it after processing). After dequeuing one, 2 messages remain. Key difference: Peek = read-only look; Dequeue = take and process (message becomes invisible to others during processing).

10. Storage design scenario.

Design: Storage Account type = StorageV2 with HNS enabled (ADLS Gen2 for analytics). Redundancy = ZRS (production data). Containers: (a) "sales-data" — daily CSV files for analytics, Access tier: Hot (frequently queried), Access: Private, Lifecycle policy: move to Cool after 90 days. (b) "product-images" — website images, Access tier: Hot (served to users), Access: Blob-level public (images need public URLs). (c) "compliance-archive" — 7-year-old documents, Access tier: Archive (cheapest, rarely accessed), Access: Private, Immutability policy: 7-year retention. Total: 1 storage account, 3 containers, mixed tiers, ZRS redundancy.

Section I: REST APIs — Answers

1. What is a REST API?

REST (REpresentational State Transfer) API is a set of rules for applications to communicate over HTTP using standard methods (GET, POST, PUT, DELETE) and URLs. Restaurant analogy: You (client) look at the menu (API docs), tell the waiter (API) what you want (request), the waiter goes to the kitchen (server), and brings back your food (response). You never go into the kitchen directly — the API is the intermediary.

2. HTTP methods and CRUD mapping.

GET = Read (no body). POST = Create (has body). PUT = Update/Replace full resource (has body). PATCH = Update partial resource (has body). DELETE = Delete (no body usually). Methods with request body: POST, PUT, PATCH. Methods without body: GET, DELETE.

3. Products API calls.

Create: POST /api/products (body with product data). Get all: GET /api/products. Get one: GET /api/products/55. Update price: PATCH /api/products/55 (body: {"price": 999}). Delete: DELETE /api/products/55.

4. Difference between 200, 201, 204.

200 OK: Request succeeded, data returned in response body (e.g., GET request returning a list). 201 Created: New resource created successfully, response often includes the created object + Location header (e.g., POST creating a customer). 204 No Content: Request succeeded but nothing to return in body (e.g., DELETE — resource deleted, nothing to send back).

5. 401 vs 403.

401 Unauthorized: No authentication provided. "Who are you? You didn't show any ID." Example: Calling Azure API without including an Authorization header/Bearer token. 403 Forbidden: Authenticated (token is valid) but not authorized for this action. "I know who you are, but you're not allowed here." Example: A user with Reader role tries to DELETE a resource — identity is confirmed, but the role doesn't permit deletion.

6. Bearer Token flow.

Step 1: App sends credentials (Client ID + Secret, or uses Managed Identity) to Entra ID token endpoint. Step 2: Entra ID validates credentials and returns an Access Token (JWT), valid for ~1 hour. Step 3: App includes the token in every API call via the Authorization header: "Authorization: Bearer eyJhbG...". Step 4: Azure service validates the token (signature, expiry, audience) and checks RBAC permissions. If valid and authorized → returns data. If token expired → 401. If no permission → 403.

7. JWT structure.

JWT (JSON Web Token) has 3 parts separated by dots: Header (algorithm + token type, e.g., RS256/JWT), Payload (claims — sub: subject/user ID, aud: audience/API resource, exp: expiry timestamp, iss: issuer like Entra ID, roles: assigned roles), Signature (cryptographic hash proving the token wasn't tampered with, signed with the issuer's private key). You can decode at jwt.ms or jwt.io.

8. Five important request headers.

1) Authorization: Carries the Bearer token for authentication. 2) Content-Type: Tells the server what format the body is in (application/json). 3) Accept: Tells the server what format you want the response in. 4) x-ms-version: Azure-specific — specifies which API version to use. 5) If-Match: Sends an ETag for optimistic concurrency — update only if the resource hasn't changed since you last read it.

9. Handling 429 Too Many Requests.

429 means you've exceeded the API's rate limit (too many calls in a short time). The response includes a Retry-After header (e.g., "Retry-After: 30") telling you how many seconds to wait. Your app should: 1) Read the Retry-After header, 2) Wait that many seconds, 3) Retry the request. Best practice: Implement exponential backoff — wait 1s, then 2s, then 4s, then 8s between retries. Never retry immediately in a loop.

10. REST vs SOAP.

Format: REST uses JSON (lightweight) vs SOAP uses XML (verbose). Protocol: REST uses HTTP only vs SOAP supports HTTP, SMTP, TCP. Speed: REST is faster (smaller payloads) vs SOAP is slower. Complexity: REST is simple and easy to learn vs SOAP is complex with strict WSDL contracts. Caching: REST supports GET caching vs SOAP does not. REST is used by 95%+ of new APIs because it's simpler, faster, and works naturally with web/mobile/cloud apps. SOAP is only found in legacy enterprise systems (banking, government).
